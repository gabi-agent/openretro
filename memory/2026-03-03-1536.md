### ❌ 1. **Search Architecture Inconsistency (CRITICAL)**

**Problem:** The implementation has TWO different search approaches that don't work together:

**Migration creates:**
```sql
-- Trigger uses ContentPlainText
NEW."SearchVector" := to_tsvector('simple',
    coalesce(NEW."Title", '') || ' ' ||
    coalesce(NEW."ContentPlainText", '')
);
```

**Query uses:**
```csharp
// Query uses Title + Content (NOT SearchVector)
EF.Functions.ToTsVector("english", t.Title + " " + t.Content)
    .Matches(EF.Functions.PlainToTsQuery("english", keyword))
```

**Impact:**
- **SearchVector column is NEVER USED** in queries
- GIN index is wasted (not utilized)
- ContentPlainText is NEVER POPULATED
- Search uses runtime computation instead of pre-computed index
- **Performance:** Every search computes tsvector on-the-fly (slow on large datasets)

**Recommendation:**
```csharp
// SHOULD BE:
query = query.Where(t =>
    t.SearchVector!.Matches(EF.Functions.PlainToTsQuery("english", keyword)));
```

---

### ❌ 2. **ContentPlainText Never Set (CRITICAL)**

**Problem:** 
- Migration adds `ContentPlainText` column
- Trigger references it for SearchVector computation
- **But ContentPlainText is never populated by application code**

**Result:** SearchVector will only index Title, not Content!

**Recommendation:**
```csharp
// Option A: In Ticket model setter
public string? Content

    set

        ContentPlainText = StripHtml(value); // Auto-populate
    }
}

// Option B: In Service before save
ticket.ContentPlainText = HtmlHelper.StripTags(ticket.Content);
```
